    248 #: Performance tuning {{{
    249 
    250 repaint_delay 10
    251 
    252 #: Delay (in milliseconds) between screen updates. Decreasing it,
    253 #: increases frames-per-second (FPS) at the cost of more CPU usage.
    254 #: The default value yields ~100 FPS which is more than sufficient for
    255 #: most uses. Note that to actually achieve 100 FPS you have to either
    256 #: set sync_to_monitor to no or use a monitor with a high refresh
    257 #: rate. Also, to minimize latency when there is pending input to be
    258 #: processed, repaint_delay is ignored.
    259 
    260 input_delay 3
    261 
    262 #: Delay (in milliseconds) before input from the program running in
    263 #: the terminal is processed. Note that decreasing it will increase
    264 #: responsiveness, but also increase CPU usage and might cause flicker
    265 #: in full screen programs that redraw the entire screen on each loop,
    266 #: because kitty is so fast that partial screen updates will be drawn.
    267 
    268 sync_to_monitor yes
    269 
    270 #: Sync screen updates to the refresh rate of the monitor. This
    271 #: prevents tearing (https://en.wikipedia.org/wiki/Screen_tearing)
    272 #: when scrolling. However, it limits the rendering speed to the

    732 #: Keyboard shortcuts {{{
    733 
    734 #: For a list of key names, see: GLFW keys
    735 #: <https://www.glfw.org/docs/latest/group__keys.html>. The name to
    736 #: use is the part after the GLFW_KEY_ prefix. For a list of modifier
    737 #: names, see: GLFW mods
    738 #: <https://www.glfw.org/docs/latest/group__mods.html>
    739 
    740 #: On Linux you can also use XKB key names to bind keys that are not
    741 #: supported by GLFW. See XKB keys
    742 #: <https://github.com/xkbcommon/libxkbcommon/blob/master/xkbcommon/xkbcommon-
    743 #: keysyms.h> for a list of key names. The name to use is the part
    744 #: after the XKB_KEY_ prefix. Note that you should only use an XKB key
    745 #: name for keys that are not present in the list of GLFW keys.
    746 
    747 #: Finally, you can use raw system key codes to map keys. To see the
    748 #: system key code for a key, start kitty with the kitty --debug-
    749 #: keyboard option. Then kitty will output some debug text for every
    750 #: key event. In that text look for ``native_code`` the value of that
    751 #: becomes the key name in the shortcut. For example:
    752 
    753 #: .. code-block:: none
    754 
    755 #:     on_key_input: glfw key: 65 native_code: 0x61 action: PRESS mods: 0x0 text: 'a'
    756 
    757 #: Here, the key name for the A key is 0x61 and you can use it with::
    758 
    759 #:     map ctrl+0x61 something
    760 
    761 #: to map ctrl+a to something.
    762 
    763 #: You can use the special action no_op to unmap a keyboard shortcut
    764 #: that is assigned in the default configuration::
    765 
    766 #:     map kitty_mod+space no_op
    767 
    768 #: You can combine multiple actions to be triggered by a single
    769 #: shortcut, using the syntax below::
    770 
    771 #:     map key combine <separator> action1 <separator> action2 <separator> action3 ...
    772 
    773 #: For example::
    774 
    775 #:     map kitty_mod+e combine : new_window : next_layout
    776 
    777 #: this will create a new window and switch to the next available
    778 #: layout
    779 
    780 #: You can use multi-key shortcuts using the syntax shown below::
    781 
    782 #:     map key1>key2>key3 action
    783 
    784 #: For example::
    785 
    786 #:     map ctrl+f>2 set_font_size 20
    787 
    788 kitty_mod ctrl+shift
    789 
    790 #: The value of kitty_mod is used as the modifier for all default
    791 #: shortcuts, you can change it in your kitty.conf to change the
    792 #: modifiers for all the default shortcuts.
